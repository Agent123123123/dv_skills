# 验证策略规划

## 概述

**目标**：针对测试点列表中的每条测试点，决定**采用哪种验证手段**来证明它，并记录决策依据。

**输出**：`verification-strategy.md` —— 测试点到验证手段的映射策略表，附带决策依据

---

## 第一阶段：输入约束收集

> **Workflow 上下文**：若本 skill 由 `verification-workflow` 编排器调用，DUT 名称、Spec、RTL、`verification-targets.md`、`verification-testpoints.md` 已由前序步骤产出并存在于全局上下文中，直接使用，**不重复询问用户**。仅对下表中全局上下文未覆盖的信息项（如 FORMAL 工具授权、覆盖率收敛目标）进行询问。
>
> 若本 skill 被**独立使用**（非 workflow 上下文），则按下表逐项收集。

在开始分析之前，必须收集以下信息。对于尚未提供的信息，主动询问。

| 信息项 | 获取方式 | 缺失时处理 |
|--------|---------|-----------|
| 测试点列表（`verification-testpoints.md` 或等效描述） | 询问用户提供 | 必须获取，否则无法完成策略映射 |
| 验证目标文档（`verification-targets.md` 或等效描述） | 询问用户提供 | 必须获取，用于确认 Golden IP 排除范围 |

| 是否有 FORMAL 工具授权（如 JasperGold、VC Formal） | 询问用户 | 无工具授权则所有测试点使用 SIM |
| 覆盖率收敛目标（各优先级的 sign-off 阈值） | 询问用户 | 建议：P0 ≥ 100%、P1 ≥ 95%、P2 ≥ 80% |

## 第二阶段：可用验证手段说明

本流程只区分**两种**验证手段：

### 手段一：动态仿真（SIM）

**定义**：通过运行 RTL 仿真驱动激励、观察响应来验证行为。

**适合**：
- 功能行为验证（需要运行设计才能观察到的行为）
- 需要参考模型 / 记分板进行数据比对的测试点
- 性能/吞吐量测量
- 状态空间大、需要覆盖率驱动的测试点
- 复杂协议交互、数据通路变换

**局限**：概率性，不能穷举；验证完备性依赖覆盖率收敛。

### 手段二：形式化验证（FORMAL）

**定义**：使用模型检查（BMC、k-归纳、PDR/IC3）对 SVA 属性进行数学穷举证明。

**适合**：
- 可表述为"对所有输入，属性 P 恒成立"的测试点
- 结构性不变式：One-hot、互斥、无死锁
- 协议握手不变式（如：VALID 拜高期间 DATA 稳定）
- 复位后状态正确性
- 随机仿真极难命中的深度角落（概率 < 1/2^20）
- DUT 子范围较小（状态位数 < ~50K）且无复杂数据变换

**局限**：状态空间爆炸，不适合大型数据通路；需 SVA 属性编写能力；需工具授权。

## 第三阶段：决策框架

为每条测试点按以下顺序判断：

### 规则 1 —— 优先考虑 FORMAL

若测试点**同时满足**以下所有条件，建议选 `FORMAL`：
1. 可以写成"对所有输入/序列，某属性成立"的命题
2. 所涉及的 DUT 子结构较小（无大型数据通路或存储阵列）
3. 不需要参考模型进行逐拍数据比对
4. 有 FORMAL 工具授权

否则，进入规则 2。

### 规则 2 —— 其余情况使用 SIM

不满足 FORMAL 条件的测试点，选 `SIM`：
- 功能行为（需要实际运行观察输出）
- 数据通路正确性（需要参考模型比对）
- 性能与吞吐量
- 大状态空间的协议场景

### 规则 3 —— P0 测试点可双重覆盖

若测试点标记为 P0，且同时满足 FORMAL 条件，**建议**分配 `FORMAL + SIM` 以获得最高置信度。如用户资源受限，可降为单一手段。

### 规则 4 —— 标记不在范围内的测试点

若测试点关联的目标被标记为"不在范围内"，将手段标记为 `N/A（不在范围内）` 并附注说明。

---

## 第四阶段：优先级分类

同时为每条测试点分配优先级。以下为**建议性**启发式规则，最终优先级须经用户确认：

| 优先级 | 含义 | 建议分配的测试点类型 |
|--------|------|---------------------|
| **P0** | 流片前必须通过。未通过则设计正确性存在风险。 | `[CLK_RST]`、`[SAFETY]`，以及用户明确指定的核心功能 |
| **P1** | 高价值。应在硅片 bring-up 前通过。 | `[FUNC]` 核心 Spec 需求、`[INTF]` 协议合规性 |
| **P2** | 建议完成。应在最终 sign-off 前完成。 | `[CORNER]`、`[PERF]`、`[LOW_PWR]`、`[INTEG]` |
| **P3** | 锦上添花。尽力完成。 | 附加场景、可选特性 |

> ⚠️ 上述规则仅为**建议**，不是强制分配。请在输出策略表后询问用户是否需要调整任何测试点的优先级，尤其是 P0 列表。

## 输出交付物格式

文件名：`verification-strategy.md`

```markdown
# 验证策略 — <DUT 名称>

## 摘要

| 验证手段 | 测试点数量 |
|----------|------------|
| FORMAL | <n> |
| SIM | <n> |
| FORMAL + SIM | <n> |
| N/A（不在范围内） | <n> |
| **合计** | **<N>** |

## 覆盖率收敛目标

| 优先级 | 功能覆盖率 sign-off 阈值 | 备注 |
|--------|--------------------------|------|
| P0 | 100% | 必须全部覆盖才可流片 |
| P1 | ≥ 95% | 建议值，经用户确认 |
| P2 | ≥ 80% | 建议值，经用户确认 |
| P3 | 尽力完成 | 无硬性要求 |

## 策略表

| 测试点 ID | 简要描述 | 类别 | 手段 | 优先级 | 决策依据 |
|----------|---------|------|------|--------|----------|
| TP-FUNC-001 | <简要> | FUNC | SIM | P1 | 需运行仿真观察数据通路输出 |
| TP-CLK_RST-001 | <简要> | CLK_RST | FORMAL | P0 | 结构性不变式，需穷举证明 |
| TP-INTF-001 | <简要> | INTF | FORMAL + SIM | P0 | P0 协议属性，双重覆盖 |

## 备注与假设
<依赖用户输入做出的决策，或权衡取舍说明>
```

## 完成确认

1. 向用户展示策略摘要表供审阅。
2. 重点展示所有 P0 测试点，并询问："以上 P0 测试点列表是否准确？是否需要升级或降级任何测试点的优先级？"
3. 询问："覆盖率收敛目标是否符合项目要求？"
4. 根据反馈调整后，保存为 `verification-strategy.md`。
