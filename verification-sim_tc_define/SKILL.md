# SIM 测试用例定义

## 概述

**目标**：针对验证策略中所有手段含 `SIM`（含 `FORMAL + SIM`）的测试点，精确定义**三元组**：

| 字段 | 定义 |
|------|------|
| **Stim（激励）** | 如何触发该测试点——描述初始条件、激励序列、关键事件及时序约束 |
| **Coverage（覆盖）** | 该测试点下系统**一定要出现**的状态——描述仿真中必须采样到的条件、值、时序 |
| **Checker（检查）** | 该测试点下系统**绝不能出现**的状态——描述必须始终为假的条件（断言的否定面） |

**输出**：`verification-sim-tc-defines.md`

## 第二阶段：三元组定义规范

### Stim（激励）

描述**如何主动触发**该测试点，应包含：

- **前置条件**：DUT 需处于何种初始状态（如：复位完成、FIFO 为空、FSM 处于 IDLE）
- **激励动作**：施加什么输入序列（信号名称、值、持续时间、顺序）
- **关键时序**：关键事件发生的先后顺序
- **是否需要随机化**：若 Stim 中存在可随机化的参数，明确列出随机化范围或约束

写法示例：
> "在复位释放后，将 `req` 拉高并保持 1 拍，同时驱动随机合法 `addr`（范围 0~255）；随后在 rand[1~4] 拍延迟后再拉低 `req`。"

### Coverage（覆盖）

描述在该测试点执行期间，仿真**必须采样到**的状态组合。

应描述：
- 哪个信号或信号组合处于何种值/状态
- 该条件在哪个时钟沿被采样
- 若涉及交叉覆盖，明确两个维度

写法示例：
> "必须采样到 `grant` 与 `req` 同时为高的周期；必须覆盖 `addr` 的低边界（0）、高边界（255）和中间值（128）三个 bin。"

### Checker（检查）

描述在该测试点执行期间，仿真**绝对不允许出现**的状态。

应描述：
- 违规条件的信号组合及时序
- 采用"若 A 则非 B"的蕴含式格式，便于直接映射到 SVA

写法示例：
> "若 `grant` 为高，则 `err` 必须为低（`|-> !err`）；`req` 未拜高时，`grant` 不得为高（`!req |-> !grant`）。"

## 第三阶段：执行步骤

1. **筛选 SIM 测试点**：从验证策略的策略表中，提取所有 **手段 = `SIM` 或 `FORMAL + SIM`** 的行，形成工作列表。

2. **逐条定义三元组**：对工作列表中每条测试点，依次基于 Spec 和 RTL 分析，填写 Stim / Coverage / Checker。

3. **信号名精确化**：三元组中所有信号名均使用 RTL 顶层端口或内部关键信号的**实际名称**（不允许使用"某信号"、"输出端口"等模糊表述）。

4. **随机化约束提炼**：若 Stim 中含有可随机化的维度，在 Coverage 中同步补充对应的 bins 覆盖需求，确保随机化有意义地命中目标空间。

5. **确认环节**：完成全部三元组草稿后，向用户显示汇总表，重点标出 P0 测试点的三元组，询问：
   > "以上三元组定义是否准确？是否需要调整任何测试点的 Stim / Coverage / Checker？"

---

## 质量检查清单

在保存输出前，逐条确认：

- [ ] 每个三元组的 Stim 中**不存在模糊信号名**（必须是 RTL 实际信号）
- [ ] Coverage 的每个 bin 与 Stim 的激励范围**相互覆盖**（激励真的能触发该 bin）
- [ ] Checker 条件与 Spec 的功能要求**直接对应**（能从 Spec 找到依据）
- [ ] P0 测试点的 Checker 至少包含**一条安全性断言**（系统级不变式）
- [ ] 不存在 Coverage 和 Checker 同时描述**同一信号同一值**的矛盾
